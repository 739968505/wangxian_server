LAMP网络传输框架介绍

1.   概述

LAMP是Light Asynchronous Message Protocol的缩写，此协议是基于TCP或者UDP上的应用层协议。旨在将TCP/UDP层的相关网络操作对开发人员屏蔽，让开发人员更加专注于应用本身，而非网络本身。

通常开发基于TCP/UDP的程序，开发人员会花费大量的时间来设计网络传输协议，同时要管理网络连接，要处理各种网络数据包的异常情况，LAMP帮助开发人员解决了这些问题，使得基于LAMP开发的应用程序变得异常的容易。

LAMP的原理：

 

 

第一步：应用程序从Lamp层中取出一个链接，根据需要设置此链接的消息处理器

第二步：将REQ请求通过此链接发送取出，此发送操作将立即返回，无需等待Lamp是否真的通过网络发送

第三步：Lamp将此REQ发送给对等体，同时缓存此REQ到滑动窗口

第四步：对等体Lamp将收到的REQ通知消息处理器

第五步：对等体的消息处理器处理此REQ，并且返回一个RES给对等体的Lamp层

第六步：对等体Lamp层将RES发送出此Lamp层

第七步：Lamp层从滑动窗口中取出对应的REQ，将REQ，RES一起通知消息处理器

 

2.   功能

Lamp具备如下的功能：

 

面向消息

Lamp层与应用层之间的接口参数都是消息，而且数据或者流。所以单个消息不能太大，否则内存开销将是一个问题，所以Lamp不适用于文件传输。

异步传输

Lamp层在接收到应用层消息后，会将消息写入到缓冲区中，并且释放应用层调用线程的资源。同样，当Lamp层收到来自于对等体Lamp的消息数据后，也是先将数据放入到一个缓冲区中，并且从线程池中取出一个线程来处理收到的消息数据。

自动管理链接

Lamp自行管理链接，应用层需要发送数据的时候，只需要向Lamp索取链接就可以了，无需关心此链接是怎么来的。当Lamp收到对等体来的消息后，自动通知应用层有消息到达。同时Lamp还会选择在空闲的时候检查链接是否可用，检查链接是否可以关闭以释放资源等。当连接数不够使用的时候，Lamp还会在允许的范围内自动增加链接数以提高传输的效率。

消息签名

Lamp层在传递消息的时候，可以根据消息的配置自动对消息进行数字签名已防止数据在传输过程中被篡改。

双工模式

两个Lamp层对等体都可以主动向对方发送数据

服务器模式

Lamp层在作为服务器模式运行的时候，支持不同的客户端来链接。一个Lamp服务器可同时支持大量的客户端链接，Lamp设计本身不会是瓶颈，瓶颈在于服务器设备的性能，我们在Dell2850上进行过100亿个包的测试，同时10000万链接，每秒5万的数据包，Lamp工作的非常好。

客户端模式

Lamp层作为客户端模式运行的时候，支持链接不同的服务器端。一个Lamp客户端可以同时支持大量的与不同服务器的链接，Lamp设计本身不会是瓶颈，瓶颈在于客户端设备的性能。

消息丢失重发

Lamp层发送出去的消息，如果没有得到对等体的回应，会自动重发此条消息，直到达到最大重发次数，并且通知应用层此条消息发送失败。

断点数据保留

当Lamp层某个链接意外中断时，如果此链接上有数据没有处理，Lamp会保留这些数据，并且将这些数据通知给应用层

自动生成协议代码

Lamp可以根据一个Xml的配置文件，自动生成消息类，应用层无需关心消息类的编码，这些都可以让Lamp自动生成，应用层只需要在Xml中定义要传输什么数据即可。

可视化监控页面

Lamp提供了各个链接的显示页面，可以看到各个链接的滑动窗口使用情况，缓冲器使用情况，链接的状态，链接上处理过的消息数目等等。

 

3.   入门指南

3.1安装Lamp

Lamp的安装非常简单，你只需要将xuanzhi-tools-1.0.jar和log4j.jar加入到你的classpath中即可。Lamp对JDK版本的要求最低是1.4.2，建议你采用1.5或者以上的版本。

xuanzhi-tools-1.0.jar是灵通最基本的一个组件包，任何项目都已经包含了这个包，log4j.jar是一个第三方的包，非常的普及，一般任何项目也已经包含了这个包。

相关的JavaDoc，可以查阅：http://219.142.193.215:8082/docs/common_tools/api/

3      开发指南

Lamp所有的实现类都是位于com.xuanzhi.tools.transport

4.1 基本概念

请求消息：接口类RequestMessage, 表示Lamp一端主动向另外一端发送请求

响应消息：接口类ResponseMessage，表示Lamp一端响应另一端的一个请求

请求与响应永远是成对出现的，他们的序列号必须是一致的，这个需要应用程序在创建ResponseMessage对象的时候来引用RequestMessage的序列号。如果序列号设置不对，那么Lamp会认为对方没有相应发出去的请求，可能是请求丢失了，那么他会再次发送请求给对方。

链接：接口类Connection，表示两个对等体之间的一个网络通路。链接是有两个Lamp对等体按需自动创建的，同时在两个对等体各自通知上层的应用程序。应用程序在收到创建了链接请求后，必须设置此链接的消息工厂以及消息处理类。否则Lamp将直接关闭链接或者丢弃所有的消息。

Selector：相当于链接池，应用程序需要链接来发送请求时，需要从Selector中取出链接，然后调用链接的writeMessage方法，注意此方法调用结束后，链接就已经被自动放入到链接池中了。也就是如果你再次调用此链接的writeMessage方法，会得到一个IllegalStateException异常。同时每一个链接一旦从连接池中取出，在没有返回连接池前，将不会被其他线程取到。也就是说，链接是单线程处理，一个链接不可能被两个线程同时处理，Lamp保证了这一点，简化了多线程同步的问题。

选择策略：当从Selector中取出链接的时候，涉及到一个策略的问题。这个策略主要是解决这样的问题：到底选择什么样的链接出来以及如果没有可用的链接怎么办？典型的策略有4种，一种是默认客户端模式SelectorPolicy.DefaultClientModelSelectorPolicy，这种策略是用于所有只链接一个服务器的情况。一种是选择标识链接创建模式SelectorPolicy.IdentitySelectorAndCreatePolicy，这种策略也是客户端模式，是用于连接了多个服务器的情况。一种是默认服务器的模式SelectorPolicy.DefaultServerModelSelectorPolicy，是用于此服务器只有一个客户端来链接；第四种是标识链接模式SelectorPolicy.IdentitySelectorPolicy，适用于多个客户端来链接的情况，每个客户端分配一个固定的标识。

3.2   配置消息原型

4.2.1 文件格式

 

消息原型的配置文件格式如下：

<prototypes version='' author=''>

<verification name='' a='' b='' c='' d=''/>

<factory name='' enable-serialize='true|false'/>

    <prototype req_name='' res_name='' req_type='' res_type=''>

        <req>

           <property name='' type='' comment=''/>

           //可以有多个property

           <object name='' type='' comment=''/>

           //可以有多个object

           <object-property name='' type='' comment=''>

               <property name='' type='' comment=''/>

               //可以有多个property

           </object-property>

           //可以有多个object-property

        </req>

        <res>

           //same as req

        </res>

    </prototype>

   

//可以有多个prototype

</prototypes>
